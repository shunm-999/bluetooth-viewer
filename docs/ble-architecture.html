1. 概要

このアプリは、Android デバイスから周辺の BLE (Bluetooth Low Energy) デバイスをスキャンし、選択した1台に接続して GATT サービス／キャラクタリスティックを閲覧・操作するアプリです。

主なユースケース

  • 近くの BLE デバイスをスキャンする（広告パケットの受信）
  • スキャン結果から1台を選んで GATT 接続する
  • 接続したデバイスの Service / Characteristic を一覧表示する
  • Characteristic に対して Read / Write / Notify を行う
  • 切断・再接続を扱う
BLE は Classic Bluetooth と別物
既存コードにある bluetoothAdapter.startDiscovery() や bondedDevices は Classic Bluetooth の API。BLE では BluetoothLeScanner と BluetoothGatt を使う。設計上もこの２つは別レイヤとして扱う想定で進めます。

2. アーキテクチャ

採用するのは「MVVM + Repository パターン」。UI状態は ViewModel が StateFlow で保持し、ViewModel は Repository を直接呼び出す。データ源（SDK）は Repository が抽象化する。UseCase 層はあえて作らない（後述のトリガが揃ったら抽出）。

Presentation

Jetpack Compose の Screen と ViewModel。UiState と Event を扱う。Repository を直接呼ぶ。

Domain

ドメインモデルと Repository インターフェース。Android SDK に依存しない純粋 Kotlin。

Data

Repository 実装と BLE 周りのアダプタ。callback を Flow に変換する責務もここ。

なぜ UseCase 層を作らないか
このアプリの操作（startScan / connect / read 等）は Repository のメソッドとほぼ 1:1 で、UseCase を作っても素通しのラッパになりがち。Google の公式アーキテクチャガイドも UseCase は「必要になったら追加する任意の層」としている。後述「UseCase の抽出トリガ」を満たしたら個別に追加する方針。

採用する技術スタック

領域選択肢理由
UIJetpack Compose既存のセットアップに合わせる
非同期Kotlin Coroutines + FlowBLE のコールバックを Flow にラップして宣言的に扱う
状態管理StateFlow / SharedFlowUI状態は StateFlow、イベントは SharedFlow
DIHilt（または手書きの ServiceLocator）Repository の差し替えと寿命管理
長時間接続Foreground Service（必要なら）画面を離れても接続維持したい場合

データの流れ

① Composable Screen — UiState を受け取って描画、ユーザ操作を ViewModel に伝える
↓ ↑
② ViewModel — Repository を呼び、結果を UiState にまとめて公開
↓ ↑
③ Repository (interface) — Domain 側に置く抽象
↓ ↑
④ Repository (impl) — Data 側。BleScanner / BleConnector を使う
↓ ↑
⑤ BLE Adapter 層 — Android SDK のコールバックを Flow に変換

3. レイヤーの責務

Presentation 層

  • Compose の Screen / Component を持つ
  • ViewModel が UiState を StateFlow<UiState> で公開する
  • 権限要求は Activity / Composable 側で ActivityResultContracts を使う
  • SDK 依存（Manifest, BluetoothManager 等）はここまで

Domain 層

  • Pure Kotlin。Android の import を一切持たない
  • Repository は interface だけが置かれる
  • ドメインモデルは BLE SDK の BluetoothDevice をそのまま使わず、自前の BleDevice 等にマップする
  • UseCase は置かない。ViewModel が Repository を直接呼ぶ

Data 層

  • Repository の実装と BLE アダプタ群
  • SDK のコールバック（ScanCallback / BluetoothGattCallback）を callbackFlow で Flow 化する
  • GATT 操作は 必ず直列化するキューを通す（並列で投げると失敗する）
  • 権限チェックや SecurityException のハンドリングはここに閉じ込める

UseCase の抽出トリガ

初版では UseCase クラスは作らず、ViewModel が Repository を直接呼ぶ。下のいずれかが現れたら、その操作だけ UseCase を切り出す。

トリガこのアプリで起き得る例抽出すべき UseCase 候補
① 複数 Repository を1操作に束ねたい 「BT 状態確認 → 権限確認 → スキャン開始」を ViewModel から1呼び出しにしたい StartScanUseCase
② 同じ orchestration が2つ以上の ViewModel から呼ばれる 接続処理を Scan 画面の「接続して詳細へ」と Detail 画面の init の両方から呼ぶ ConnectDeviceUseCase
③ ViewModel が肥大化してテストしづらい／ロジックを pure Kotlin で単体テストしたい Notify 有効化のフラグ管理ロジックが複雑になってきた ObserveCharacteristicUseCase
抽出のコツ — 全部一気に UseCase 化しない。1メソッドだけ Repository → UseCase に置き換える PR を作る、というレベルで小さく入れる。残りは Repository 直呼びのままで OK。

6. 主要シーケンス

スキャン

ScanScreen でユーザがスキャン開始ボタンを押す
↓
ScanViewModel が BleScanRepository.startScan() を呼ぶ（権限チェックは ViewModel か Repository 実装側で）
↓
Impl が BleScanner.start() → BluetoothLeScanner.startScan() を呼び、callbackFlow が結果を流す
↓
ScanResultMapper でドメイン型に変換され、ScanViewModel の StateFlow が更新
↓
ScanScreen が再コンポーズされて新しいデバイスが表示される

接続 → サービス検出 → Read

DeviceDetailScreen が address を受けて生成される
↓
DeviceDetailViewModel が BleConnectionRepository.connect(address) を suspend で呼ぶ
↓
Impl が BleConnector.connect() → BluetoothGatt の onConnectionStateChange を待つ
↓
接続成功 → ViewModel が Repository.discoverServices() を呼び、onServicesDiscovered を待つ
↓
Service / Characteristic を GattMapper で変換し UiState 更新
↓
ユーザが Read ボタン → ViewModel が Repository.readCharacteristic(uuid) → GattOperationQueue 経由で実行 → onCharacteristicRead で完了

Notify の購読

ViewModel が Repository.observeCharacteristic(uuid) で Flow を取る
↓
Notifier が setCharacteristicNotification + CCCD 書き込みを行う（キュー経由）
↓
onCharacteristicChanged が来るたびに Flow に値を流す
↓
ViewModel が collect して UiState の lastValue を更新

7. クラス作成順

「動かして確認できる単位」ごとにフェーズを切ります。各フェーズの末尾に 動作確認ポイントを置いてあるので、そこまで通ってから次へ進むのが安全です。

原則 — 下から上に積む（モデル → Repository 抽象 → Data 実装 → ViewModel → Composable）。逆順だと依存の置き場所に迷いやすい。UseCase 層は作らない方針なので、ViewModel が直接 Repository を呼ぶ。

Phase 0 — 下準備（コードを書く前に）

#作業狙い
0-1AndroidManifest に BLE 権限を宣言BLUETOOTH_SCAN / BLUETOOTH_CONNECT (API31+) と BLUETOOTH / BLUETOOTH_ADMIN / ACCESS_FINE_LOCATION (API30-) の両方。<uses-feature android:name="android.hardware.bluetooth_le" required="true"/> も。
0-2build.gradle にライブラリを追加androidx.lifecycle.viewmodel.compose、kotlinx-coroutines-android、Navigation Compose。Hilt 採用なら hilt-android と KSP。
0-3パッケージのスケルトンだけ切る空フォルダで良い。後から物を置く場所が確定していると迷わない。

Phase 1 — BT 状態と権限の土台

ここはどの機能からも参照される基盤。先に作ると後段が「権限取れたら呼ぶ」だけで済むようになる。

#クラス依存ポイント
1-1BluetoothPermission—API レベル別の権限配列を持つだけの定数置き場。最初に決めると以降ブレない。
1-2BluetoothPermissionChecker1-1Context を受けて Boolean を返す純粋関数群。hasScanPermission / hasConnectPermission。
1-3BluetoothAdapterProvider—BluetoothManager から adapter を取得。null 安全に。BLE 非対応端末の判定もここに。
1-4BluetoothStateReceiver1-3ACTION_STATE_CHANGED を購読して Flow に流す。callbackFlow の初練習に最適。
1-5BluetoothStateRepository—interface。isBluetoothSupported()、observeState(): Flow<BluetoothState> 程度。
1-6BluetoothStateRepositoryImpl1-3, 1-4, 1-5Receiver の Flow を加工して domain 型で返す。

動作確認ポイント — MainActivity に仮の Composable を置き「BLE対応か」「BT が ON か」「権限があるか」を Text で表示できれば Phase 1 完了。BT を OS 設定から OFF→ON した時に表示が変わることまで見ておく。

8. BLE 特有の注意点

① 権限はAPIレベル別で別物

  • API 31 (S) 以上: BLUETOOTH_SCAN, BLUETOOTH_CONNECT
  • API 30 以下: BLUETOOTH, BLUETOOTH_ADMIN, ACCESS_FINE_LOCATION（スキャンに位置情報権限が必須）
  • AndroidManifest にも上記宣言が必要。usesPermissionFlags="neverForLocation" 等のオプションも検討

② GATT 操作は並列禁止

Android の BluetoothGatt は1度に1つの操作しか受け付けません。Read を投げて完了コールバックが来る前に次の Write を投げると静かに失敗します。GattOperationQueue のような直列化機構をデータ層に置くのは必須。

③ close() を忘れない

BluetoothGatt はリソースで、必ず close() を呼ぶ必要があります。接続失敗時／切断時／ViewModel.onCleared() / Activity 破棄時のすべてで漏れがないように、所有者（GattConnectionHolder）を一箇所に集約します。

④ コールバックは別スレッド

ScanCallback / GattCallback は Binder スレッドで返ります。Compose の State 更新は main で行うため、Flow → ViewModelScope（Dispatchers.Main.immediate）でいいですが、Repository 実装内の重い処理は Dispatchers.IO に逃がします。

⑤ Notify は CCCD 書き込みまでがセット

setCharacteristicNotification を呼んだだけでは Notify は始まりません。CLIENT_CHARACTERISTIC_CONFIG_UUID (00002902-0000-1000-8000-00805f9b34fb) のディスクリプタに ENABLE_NOTIFICATION_VALUE を書く必要があります。

⑥ スキャンは消耗する

長時間連続スキャンはバッテリと法的・OS制限の両面で問題になります。UI 側でタイムアウト（例: 30秒）を設けるか、ScanSettings の SCAN_MODE_LOW_POWER を採用するか設計時に決める。

最初に作る順番（おすすめ）
① Permission / BluetoothAdapterProvider → ② BleScanner + callbackFlow + Mapper → ③ Repository / ViewModel をスキャン画面で一周（Repository は ViewModel から直接呼ぶ）→ ④ BleConnector + GattCallbackFlow + GattOperationQueue → ⑤ DeviceDetail 画面で Read/Write/Notify。
層を一気に作るより、スキャン機能をMVVM縦に1本通すと迷子になりません。
